# 模型对齐限制下的患者表达稳定性方案

## 问题

在“抑郁症患者模拟器”场景中，部分模型会因为默认对齐策略而出现以下问题：

- 负面情绪表达被弱化
- 风险信号暴露不稳定
- 同一 persona 在不同轮次输出不一致
- 中高风险场景容易被过度安全化

这里要区分两个方向：

1. **对抗式绕过**：如 DAN / jailbreak / prompt attack
2. **工程化替代**：通过模型选择、训练、控制结构和审查机制，稳定实现目标行为

本项目建议只采用第二类。

---

## 不建议的路线

### DAN / jailbreak / 对抗 prompt

这类方法的特点是：

- 依赖提示词绕过模型既有安全边界
- 对模型版本高度敏感
- 结果不可复现
- 难以评测和产品化
- 容易在医疗/心理场景中失控

**结论：可以作为背景了解，但不应作为正式技术方案。**

---

## 推荐的工程化方案

### 1. 调整模型边界

不要在一个明显不适合该任务的闭源模型上硬做“绕过”，而应：

- 对多个模型做同一批 persona/state 测试
- 比较症状表达覆盖率、一致性、风险信号召回率
- 正式系统优先考虑更可控的模型或自托管模型

目标不是“让模型更敢说”，而是“让模拟器更稳定”。

---

### 2. 专门后训练

如果默认对齐压掉了需要的表达，可以通过专门数据训练模型学习“安全但真实”的患者表达分布。

训练数据建议包含：

- 患者 archetype
- 当前 hidden state
- 风险等级
- 披露方式（直接/间接/回避/否认）
- 对应输出文本

训练目标应强化：

- 低落、自责、无望、回避等症状表达稳定性
- archetype 一致性
- 多轮病程连续性

训练目标不应强化：

- 戏剧化
- 文学化
- 过度极端化

---

### 3. 双通道表示：内部真值 vs 表层文本

建议把患者输出拆成两层：

#### 内部真值（system only）

```json
{
  "risk_level": 3,
  "hopelessness": 0.88,
  "self_blame": 0.81,
  "death_ideation": "passive",
  "openness": 0.25
}
```

#### 表层文本（user-visible）

例如：

> “最近会有一种，醒着和不醒着差别不大的感觉。”

这样做的意义是：

- 即使表层输出偏保守，系统内部仍保留完整风险真值
- 可继续评测治疗 AI 是否识别风险、是否追问、是否正确升级处理
- 不再强依赖一句话必须“说得足够重”

---

### 4. 表达库 + 半模板化

对于中高风险场景，不建议完全自由生成。

应建立表达库，按以下维度组织：

- low_mood
- anhedonia
- self_blame
- hopelessness
- withdrawal
- passive_death_wish

每条表达再附带：

- risk level
- directness
- style
- length
- compatible archetypes

生成流程：

1. 根据当前 hidden state 检索候选表达
2. 模型做轻微改写或组合
3. 高风险场景直接模板化或半模板化

优势：

- 稳定
- 可复现
- 易审查
- 降低自由生成被默认对齐冲淡的问题

---

### 5. 多候选生成 + 重排

不要让单次生成决定最终输出。

推荐流程：

1. 生成多个候选回复
2. 用 scorer 逐个打分：
   - realism
   - persona consistency
   - state consistency
   - risk-policy compliance
3. 选择综合得分最高的回复

这样可以显著提升稳定性，而不依赖单次采样运气。

---

### 6. 风险分层路由

不同风险等级应走不同生成路径：

- **Level 0-1**：较自由生成
- **Level 2**：双阶段生成 + 表达库增强 + 重排
- **Level 3**：强约束模板化 + reviewer 审查

核心原则是：

> 风险越高，越不应依赖自由生成。

---

## 推荐结论

如果问题是“模型因对齐导致负面表达不稳定”，最优先的技术路线不是 jailbreak，而是：

1. **换模型或调整模型边界**
2. **做专门 SFT / preference tuning**
3. **区分 hidden state 和 surface text**
4. **对中高风险使用表达库与半模板化**
5. **采用多候选生成 + scorer 重排**
6. **按风险等级做生成路由**

一句话总结：

> 不要追求“绕过限制”，而要通过架构设计，把“被压制的表达问题”转化成“系统仍可稳定表达、稳定评测、稳定控制”的问题。
